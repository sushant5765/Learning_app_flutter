1.1 BACKEND DESIGN

1.1.1 AI Model Architecture
----------------------------
The application implements two custom-trained deep learning models for natural 
language processing tasks:

A. SUMMARIZATION MODEL
   - Architecture: LSTM-based Sequence-to-Sequence (Seq2Seq) model
   - Encoder: Bidirectional LSTM (128 hidden dimensions)
   - Decoder: Unidirectional LSTM (128 hidden dimensions)
   - Embedding Layer: 128-dimensional word embeddings
   - Vocabulary Size: 5,002 tokens (including <PAD> and <UNK>)
   - Input Length: Maximum 200 tokens
   - Output Length: Maximum 50 tokens
   - Total Parameters: ~6.4 MB (PyTorch model)
   - Mobile Format: TensorFlow Lite (383 KB compressed)

B. QUESTION ANSWERING MODEL
   - Architecture: Siamese LSTM Network
   - Question Encoder: Bidirectional LSTM (128 hidden dimensions)
   - Context Encoder: Bidirectional LSTM (128 hidden dimensions)
   - Answer Decoder: Unidirectional LSTM (256 hidden dimensions)
   - Similarity Layer: Linear layer for question-context matching
   - Vocabulary Size: 5,002 tokens
   - Input Lengths: Question (30), Context (200), Answer (50)
   - Total Parameters: ~11 MB (PyTorch model)
   - Mobile Format: TensorFlow Lite (542 KB compressed)

1.1.2 Model Training Pipeline
-----------------------------
The training process follows a structured pipeline:

Step 1: Data Preparation (01_prepare_data.py)
  - Loads raw text data from CSV files
  - Cleans and normalizes text (removes special characters, normalizes whitespace)
  - Creates extractive summaries (first 2-3 sentences)
  - Generates Q&A pairs from text passages
  - Output: Structured CSV files for summarization and Q&A tasks

Step 2: Feature Engineering (02_feature_engineering.py)
  - Text tokenization using NLTK
  - Vocabulary building (top 5,000 most common words)
  - Sequence conversion (text → integer sequences)
  - Feature extraction (word count, sentence count, average lengths, etc.)
  - Padding/truncation to fixed lengths
  - Output: Vocabulary JSON, sequence CSV files, feature CSV files

Step 3: Model Training (03_train_summarizer.py, 04_train_qa.py)
  - Data splitting: 80% training, 20% validation
  - Batch size: 16
  - Optimizer: Adam (learning rate: 0.001)
  - Loss function: CrossEntropyLoss (ignoring padding tokens)
  - Training epochs: 20 (summarizer), 15 (Q&A)
  - Gradient clipping: 1.0 (prevents exploding gradients)
  - Teacher forcing: Used during training for faster convergence
  - Model checkpointing: Saves best models to disk

1.1.3 Model Conversion Pipeline
--------------------------------
PyTorch → TensorFlow Lite Conversion:
  - Trained PyTorch models (.pth files) are converted to TFLite format
  - Simplified architecture for mobile compatibility (TFLITE_BUILTINS only)
  - Model quantization: Float32 (maintains accuracy)
  - Final models stored in assets/models/ for Flutter integration

1.1.4 Service Architecture
---------------------------
The application uses a layered service architecture:

A. TfliteModelService (lib/services/tflite_model_service.dart)
   - Loads TFLite models from assets
   - Manages vocabulary (word-to-index and index-to-word mappings)
   - Handles text preprocessing (tokenization, sequence conversion)
   - Performs model inference (summarization and Q&A)
   - Error handling and fallback mechanisms

B. AiModelService (lib/services/ai_model_service.dart)
   - High-level interface for AI model operations
   - Routes requests to TFLite models (mobile) or Python scripts (desktop)
   - Platform-specific logic (iOS, Android, Desktop)

C. AiService (lib/services/ai_service.dart)
   - Main AI service orchestrator
   - Manages conversation history
   - Implements fallback to LocalNlpService when models fail
   - Caches summaries and highlights for performance

D. LocalNlpService (lib/services/local_nlp_service.dart)
   - Rule-based NLP fallback system
   - Extractive summarization (sentence ranking)
   - Keyword extraction and vocabulary analysis
   - Question answering using keyword matching

1.1.5 Data Storage
------------------
- Local Storage: Hive (NoSQL database) for user preferences and history
- Model Storage: Assets folder (bundled with app)
- Vocabulary: JSON file (5,002 word mappings)

4.2 COMPARATIVE ANALYSIS
------------------------

4.2.1 Trained Models vs. Fallback System
-----------------------------------------

Trained Models (TFLite):
  Advantages:
    - Context-aware responses
    - Better understanding of semantic relationships
    - More natural language generation
    - Learns from training data patterns

  Limitations:
    - Requires model loading (1-2 seconds)
    - Fixed vocabulary (5,002 words)
    - Limited input/output lengths
    - Platform compatibility issues (iOS)

Fallback System (LocalNlpService):
  Advantages:
    - Always available (no loading required)
    - Works on all platforms
    - No memory overhead for models
    - Fast response time

  Limitations:
    - Rule-based (less intelligent)
    - Keyword matching approach
    - Less context understanding
    - Simpler responses


4.3 MODEL LIMITATIONS AND FUTURE IMPROVEMENTS
----------------------------------------------

4.3.1 Current Limitations
-------------------------
1. Vocabulary Size: Limited to 5,002 words
   - Impact: Unknown words mapped to <UNK> token
   - Solution: Increase vocabulary or use subword tokenization

2. Sequence Length: Fixed maximum lengths
   - Impact: Long texts truncated, short texts padded
   - Solution: Dynamic sequence handling or hierarchical models

3. Model Architecture: Simple LSTM-based
   - Impact: May miss long-range dependencies
   - Solution: Transformer-based models (BERT, GPT) for better performance

4. Training Data: Limited dataset size
   - Impact: Model may not generalize to all domains
   - Solution: Collect more diverse training data

5. iOS Compatibility: TFLite models don't load
   - Impact: iOS users get fallback system
   - Solution: Investigate tflite_flutter iOS compatibility or use Core ML

4.3.2 Potential Improvements
----------------------------
1. Model Architecture:
   - Upgrade to Transformer-based models (better accuracy)
   - Use attention mechanisms for better context understanding
   - Implement hierarchical encoding for long documents

2. Training:
   - Increase training data diversity
   - Use transfer learning (pre-trained models)
   - Implement fine-tuning on domain-specific data

3. Deployment:
   - Fix iOS compatibility (Core ML conversion or tflite_flutter update)
   - Implement model quantization (INT8) for smaller size
   - Add model versioning and update mechanism

4. Evaluation:
   - Implement quantitative metrics (BLEU, ROUGE scores)
